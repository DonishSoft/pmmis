## 1. Введение

Настоящий отчёт представляет результаты работ по разработке информационной системы **PMMIS** (Project Management & Monitoring Information System) в рамках проекта **WSIP** (Water Supply and Sanitation Investment Project), финансируемого Всемирным Банком.

Система разработана в соответствии с Финальной Концепцией Системы (обновлённая версия, ноябрь 2025) и призвана обеспечить прозрачность, дисциплину и контроль при реализации инвестиционных проектов водоснабжения в Республике Таджикистан.

Согласно ТЗ, центральная логика PMMIS строится вокруг ключевой управленческой цепочки:

> **Контракт → АВР → Платёж → Обязательства → Остаток**

Эта цепочка отражает реальный управленческий процесс ЦУП и PMU и является основной зоной риска при реализации проектов. PMMIS **не является ERP**, не заменяет бухгалтерию и не предназначена для полной автоматизации процессов учреждения.

## 2. Назначение и цели системы

### 2.1. Основное назначение

PMMIS — это **лёгкая, практичная система управления исполнением контрактов**, АВР, платежами и закупками, с возможностью учёта технических contract-linked индикаторов и ведением базовой отчетности по WB-индикаторам.

### 2.2. Ключевые цели

| № | Цель | Статус |
|---|------|--------|
| 1 | Обеспечение прозрачности цепочки «Контракт → АВР → Платёж → Остаток» | ✅ Реализовано |
| 2 | Автоматический расчёт KPI по контрактам, закупкам и платежам | ✅ Реализовано |
| 3 | Управление contract-linked индикаторами (автоматический расчёт) | ✅ Реализовано |
| 4 | Ведение outcome-level индикаторов WB (ручной ввод) | ✅ Реализовано |
| 5 | Мониторинг рисков через «Управленческие сигналы» (красные флаги) | ✅ Реализовано |
| 6 | Минимальная нагрузка на пользователей | ✅ Реализовано |
| 7 | Многоязычность (русский, таджикский, английский) | ✅ Реализовано |
| 8 | Экспорт в Excel/PDF | ✅ Реализовано |

## 3. Архитектура системы

### 3.1. Архитектурный подход — Clean Architecture

Система построена на принципах **Clean Architecture** (Чистая Архитектура), обеспечивающих разделение ответственности, тестируемость и масштабируемость.

```
┌─────────────────────────────────────────────────────────┐
│                     PMMIS.Web                           │
│         (MVC Controllers, Views, ViewModels)            │
│       Razor Views + JavaScript + Syncfusion EJ2         │
├─────────────────────────────────────────────────────────┤
│                  PMMIS.Application                      │
│          (Services, Interfaces, DTOs)                   │
│        Бизнес-логика и сервисные операции                │
├─────────────────────────────────────────────────────────┤
│                 PMMIS.Infrastructure                    │
│       (DbContext, Migrations, Identity, Repos)          │
│           PostgreSQL через Npgsql EF Core               │
├─────────────────────────────────────────────────────────┤
│                    PMMIS.Domain                         │
│         (Entities, Enums, Business Rules)               │
│           Ядро системы — доменная модель                 │
└─────────────────────────────────────────────────────────┘
```

### 3.2. Описание слоёв

| Слой | Назначение | Ключевые компоненты |
|------|-----------|---------------------|
| **PMMIS.Domain** | Ядро системы — сущности и бизнес-правила | 28 сущностей: `Contract`, `Payment`, `ProcurementPlan`, `ProjectTask`, `Indicator`, `Geography`, `Facility` и др. |
| **PMMIS.Application** | Бизнес-логика и интерфейсы сервисов | Interfaces, DTOs, Service contracts |
| **PMMIS.Infrastructure** | Инфраструктура доступа к данным | EF Core DbContext (Npgsql), Migrations, ASP.NET Identity |
| **PMMIS.Web** | Пользовательский интерфейс | 20 контроллеров, Razor Views, JavaScript, Syncfusion EJ2 |

### 3.3. Базовые паттерны доменной модели

Все сущности системы наследуют от базовых классов, обеспечивая единообразие:

- **BaseEntity** — минимальное ядро с полями `Id`, `CreatedAt`, `UpdatedAt`
- **LocalizedEntity** — поддержка трёхъязычности (`NameRu`, `NameTj`, `NameEn`) с утилитарным методом `GetName(lang)`

## 5. Модульная структура PMMIS

Система состоит из **14 функциональных модулей**, организованных в три категории:

```
PMMIS
├── ОСНОВНОЕ (Core)
│   ├── Dashboard          — Панель управления и KPI
│   ├── Контракты          — Реестр контрактов
│   ├── Подрядчики         — Реестр подрядчиков
│   ├── Проекты            — Структура проектов и компонентов
│   ├── План закупок       — Управление закупками
│   ├── Платежи            — Управление платежами
│   ├── Прогресс работ     — Физический прогресс по контрактам
│   └── АВР               — Реестр актов выполненных работ
│
├── УПРАВЛЕНИЕ (Management)
│   ├── Задачи             — Управление задачами (Kanban/Calendar)
│   └── Уведомления        — Многоканальные уведомления
│
└── СПРАВОЧНИКИ (Reference Data)
    ├── География          — Районы, джамоаты, сёла
    ├── Индикаторы / KPI   — Индикаторы проекта
    ├── Справочники        — Категории и типы

```

## 6. Описание модулей

### 6.1. Dashboard (Панель управления)

**Контроллер:** `DashboardController`

Dashboard — это центральная панель управления, обеспечивающая мгновенный обзор состояния проекта. При входе пользователь видит агрегированные показатели и критические сигналы.

#### Основные компоненты:

**KPI-карточки верхнего уровня:**
| Карточка | Описание |
|----------|----------|
| Общий бюджет | Суммарный бюджет проекта (напр. $25 165 740) |
| Сумма контрактов | Суммарная стоимость всех контрактов |
| Выплачено | Общая сумма оплаченных платежей с процентом освоения |
| Средний прогресс | Средний процент физического выполнения по контрактам |

**KPI-карточки второго уровня:**
| Карточка | Описание |
|----------|----------|
| Закупки | Количество и объём активных закупок |
| К оплате | Сумма ожидающих/одобренных платежей |
| Просрочено | Количество контрактов с просроченными этапами |
| Завершено закупок | Прогресс завершённых закупочных процедур |

**Управленческие сигналы (Красные флаги):**

Система автоматически вычисляет и отображает критические управленческие сигналы на Dashboard:
- 🔴 Просроченные этапы контрактов (с указанием количества дней просрочки)
- 🔴 Просроченные задачи (проверка АВР, подготовка документов)
- ⚠️ Задержки платежей
- ⚠️ Задержки закупок

Каждый сигнал содержит ссылку на конкретный объект, что позволяет быстро перейти к проблемному контракту или задаче.

### 6.2. Модуль «Проекты»

**Контроллер:** `ProjectsController`

Модуль обеспечивает иерархическое управление структурой проектов по стандартам Всемирного Банка.

#### Иерархия:
```
Проект (Project)
  └── Компонент (Component)
       └── Подкомпонент (SubComponent)
```

#### Функциональность:
- **Создание и редактирование проектов** — код проекта, название, бюджет, сроки
- **Управление компонентами** — каждый компонент имеет свой бюджет и описание
- **Привязка контрактов** — контракты привязываются к компонентам проекта
- **Автоматический расчёт** — общий бюджет агрегируется из компонентов
- **Мультиязычность** — названия на русском, таджикском и английском

1. Инвестиции в объекты ВКХ Вахшской межрайонной системы ($23 605 880)
2. Создание потенциала и изменение поведения ($1 559 860)
3. Управление проектом и поддержка реализации

### 6.3. Модуль «Контракты»

**Контроллер:** `ContractsController`

Модуль «Контракты» — это **ядро системы PMMIS**, вокруг которого строится вся управленческая цепочка. Реестр контрактов обеспечивает полную прозрачность исполнения.

#### Основные ата каждого контракта:
| Поле | Описание |
|------|----------|
| Номер контракта | Уникальный идентификатор (напр. RWSSP-W/006-02 Lot2) |
| Объём работ | Описание работ по контракту |
| Подрядчик | Привязка к реестру подрядчиков |
| Тип контракта | Работы / Консалтинг |
| Дата подписания | Дата подписания контракта |
| Срок завершения | Плановый срок завершения |
| Сумма ($) | Стоимость контракта в USD |
| Прогресс (%) | Процент физического выполнения |
| Оплачено (%) | Процент оплаченной суммы |

#### Финансовая модель контракта:
```
Начальная сумма контракта
  + Сумма дополнений (Amendments)
  - Экономия (Savings)
  ────────────────────────
  = Итоговая сумма контракта
  - Оплачено (Paid)
  ────────────────────────
  = Остаток по контракту
```

#### Дополнительные возможности:
- **Этапы контракта (Milestones)** — для контроля промежуточных дедлайнов
- **Дополнения к контракту (Amendments)** — фиксация изменений суммы или сроков
- **Привязка индикаторов** — contract-linked индикаторы для автоматического расчёта прогресса
- **Мониторинг** — визуальный мониторинг состояния всех контрактов
- **Фильтрация** — по типу контракта, статусу, подрядчику, текстовый поиск
- **Экспорт** — Excel Export и PDF Export

### 6.5. Модуль «АВР» (Акты выполненных работ)

**Контроллер:** `WorkProgressReportsController`

Реестр АВР — это централизованное хранилище всех актов выполненных работ по всем контрактам. Модуль обеспечивает полную видимость хода работ.

#### KPI-панель:
| Показатель | Описание |
|-----------|----------|
| Всего АВР | Общее количество зарегистрированных актов |
| Контрактов | По скольким контрактам есть АВР |
| Средний прогресс | Средний процент выполнения |

#### Данные каждого АВР:
- Дата составления
- Контракт (ссылка)
- Подрядчик
- Проект
- Прогресс (progress bar с процентом)
- Количество приложенных документов
- Действия: Просмотр, Редактирование, Удаление

#### Фильтрация:
- По проекту
- По контракту
- По диапазону дат (от — до)

#### Экспорт данных:
- Excel Export
- PDF Export

#### Автоматизация:
При создании АВР система автоматически:
1. Обновляет процент физического выполнения контракта
2. Создаёт задачу «Проверить АВР по контракту [номер]» для ответственного сотрудника
3. Отправляет уведомление куратору контракта

### 6.6. Модуль «Прогресс работ»

**Контроллер:** `WorkProgressController`

Модуль предоставляет визуальную карточную раскладку прогресса работ по контрактам. В отличие от табличного реестра АВР, данный модуль отображает данные в формате **карточек** (Bootstrap Cards) с визуальными прогресс-барами.

#### Данные каждой карточки:
- Номер контракта (ссылка)
- Подрядчик
- Статус: `Черновик` / `Утверждён`
- Дата
- Прогресс (progress bar)
- Описание выполненных работ
- Действия: Редактировать, На проверку, Удалить

#### Workflow (Рабочий процесс):
```
Черновик  ──►  На проверку  ──►  Утверждён
```

#### Создание нового отчёта:
Форма создания отчёта включает:
- Выбор контракта
- Указание процента прогресса
- Описание выполненных работ
- Ввод прогресса по привязанным индикаторам (с прогресс-барами для каждого индикатора)
- Загрузка подтверждающих документов

### 6.7. Модуль «Платежи»

**Контроллер:** `PaymentsController`

Модуль управления платежами реализует полный workflow одобрения платежей в соответствии с требованиями Всемирного Банка.

#### Финансовая панель:
| Показатель | Описание |
|-----------|----------|
| Всего | Общая сумма зарегистрированных платежей |
| Оплачено | Сумма фактически оплаченных платежей |
| Ожидает | Сумма платежей, ожидающих одобрения (выделено красным) |

#### Типы платежей:
- **Аванс** — предоплата по контракту
- **Промежуточный** — оплата по АВР
- **Финальный** — финальный расчёт
- **Удержание** — гарантийное удержание

#### Workflow одобрения:
```
  Ожидает  ──✓──►  Одобрен  ──►  Оплачен
     │
     └──✗──►  Отклонён
```

Для каждого платежа в статусе «Ожидает» отображаются кнопки действий:
- ✅ Одобрить
- ❌ Отклонить (с указанием причины)
- ✏️ Редактировать
- 🗑️ Удалить

#### Бизнес-валидация:
- **Hard Block** — запрет оплаты сверх бюджета контракта
- **Предупреждение 90%** — сигнал при достижении 90% суммы контракта
- **Missing AVR** — предупреждение при отсутствии АВР для промежуточного платежа
- **Аудит** — автоматическая фиксация кто и когда одобрил/отклонил платёж

### 6.9. Модуль «Индикаторы / KPI»

**Контроллер:** `IndicatorsController`

Модуль индикаторов реализован в точном соответствии с ТЗ и разделён на **два класса** индикаторов:

#### A. Contract-linked индикаторы (автоматизируемые)

Технические индикаторы, привязанные к контрактам:
- Количество установленных водомеров
- Количество построенных РЧВ / водозаборов
- Проложенные километры трубопровода
- Количество восстановленных скважин

**Механизм:** Через сущность `ContractIndicator` индикатор привязывается к контракту. При создании АВР пользователь указывает прогресс по каждому привязанному индикатору. Система автоматически обновляет достигнутое значение (`AchievedValue`) и рассчитывает процент выполнения.

#### B. Outcome-level индикаторы (ручной ввод)

Индикаторы результатов WB (PDO, intermediate outcomes):
- People with access to improved water services
- Continuity of water supply
- Non-revenue water (NRW)
- Cost recovery
- Beneficiary satisfaction

**Механизм:** Вводятся вручную через интерфейс внесения значений. Система хранит историю и позволяет отслеживать динамику.

#### Иерархическая структура:
```
Категория (PDO / Промежуточные / Contract-linked)
  └── Индикатор (с кодом, напр. PDO-1, IR-1)
       └── Подиндикатор (напр. PDO-1.1)
```

#### Типы измерения:
- **Число** — количественное значение
- **Процент** — процентное значение
- **Да/Нет** — бинарный индикатор

В системе зарегистрировано **18 индикаторов** с категориями «Цели развития проекта (PDO)» и «Промежуточные индикаторы результата».

![Модуль «Индикаторы / KPI» — иерархический список с категориями, типами и целевыми значениями](/Users/mirqurbonnaimov/.gemini/antigravity/brain/6dcbc87c-3bb8-4347-9236-729e9a7d826d/indicators_page_1771814454553.png)

### 7.2. Автоматические процессы

Система выполняет следующие автоматические действия без участия пользователя:

1. **Обновление прогресса контракта** при создании/изменении АВР
2. **Обновление значений индикаторов** при указании прогресса в АВР
3. **Создание задач** при создании АВР и платежей
4. **Отправка уведомлений** при смене статусов
5. **Расчёт управленческих сигналов** при каждом обращении к Dashboard
6. **Проверка дедлайнов** через фоновую службу `DeadlineNotificationService`
7. **Доставка уведомлений** через фоновую службу `NotificationQueueService`
